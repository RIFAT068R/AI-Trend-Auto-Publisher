# Implementation Plan: Gemini-Powered /api/generate

## Objective
Use Gemini API in `/api/generate` and return only structured JSON fields:

```json
{
  "hook": "...",
  "caption": "...",
  "hashtags": ["..."],
  "category": "...",
  "imagePrompt": "..."
}
```

If Gemini fails for any reason, return mock fallback with the same shape.

## Current State
- `src/app/api/generate/route.ts` currently returns `GeneratedPostPreview` (includes extra fields).
- `src/lib/ai.ts` currently provides only mock generation.
- No Gemini SDK or REST call integration yet.
- Existing codebase has reliable fallback pattern and error-safe route wrappers.

## Implementation Plan

### 1) Add Dedicated Generate Response Type
- In `src/lib/types.ts`, add a new type for API output:
  - `GeneratedMetadata` with exactly:
    - `hook: string`
    - `caption: string`
    - `hashtags: string[]`
    - `category: string`
    - `imagePrompt: string`
- Keep existing `GeneratedPostPreview` for pipeline internals to avoid breaking other flows.

### 2) Implement Gemini Metadata Generator in `src/lib/ai.ts`
- Add a function for Gemini-backed metadata generation, e.g.:
  - `generateMetadata(topic?: string): Promise<GeneratedMetadata>`
- Resolve key from server env only:
  - `GEMINI_API_KEY`
- Call Gemini REST endpoint using `fetch` (no extra SDK needed):
  - `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent?key=...`
- Prompt Gemini to return strict JSON only with required fields.
- Parse response safely:
  - extract text
  - parse JSON
  - validate required fields and types
- On parse/API/validation failure, fallback to mock metadata.

### 3) Add Strong Fallback Function
- In `src/lib/ai.ts`, keep a deterministic local fallback:
  - simple hook variants
  - short caption
  - hashtags default (e.g. `#AI`, `#Tech`, `#Automation`)
  - category default `AI Tools`
  - image prompt default `futuristic AI technology background`
- Ensure fallback always returns a valid `GeneratedMetadata` object.

### 4) Update `/api/generate` Route Contract
- In `src/app/api/generate/route.ts`:
  - switch GET/POST handlers to return `ApiResponse<GeneratedMetadata>`
  - keep topic extraction as-is
  - call new `generateMetadata(topic)`
- Ensure route response body is exactly:
  - `ok + data` envelope with the 5 metadata fields only in `data`
- Keep `try/catch` and return fallback-driven success where possible.

### 5) Keep Pipeline Compatibility
- Do **not** break existing pipeline usage that depends on `generatePostPreview`.
- Option A (preferred):
  - Keep `generatePostPreview` unchanged for pipeline internals.
  - Add separate Gemini metadata function just for `/api/generate`.
- This isolates API contract changes from pipeline and storage flows.

### 6) Validation and Safety
- Validate all required fields before returning Gemini output:
  - missing/invalid fields trigger fallback
- Normalize hashtags:
  - ensure array of strings
  - dedupe/trim

### 7) Testing Checklist
- `GET /api/generate?topic=...` returns only required fields.
- `POST /api/generate` with `{ topic }` returns only required fields.
- With invalid/missing key, route still returns fallback metadata.
- `npm run lint`, `npm run typecheck`, `npm run build` pass.

## File-Level Change Map
- `src/lib/types.ts`
  - add `GeneratedMetadata`
- `src/lib/ai.ts`
  - add Gemini request + parser + fallback metadata function
- `src/app/api/generate/route.ts`
  - switch response type to `GeneratedMetadata` and call new function

## Done Criteria
- `/api/generate` uses Gemini when available.
- Response payload contains only:
  - `hook`
  - `caption`
  - `hashtags`
  - `category`
  - `imagePrompt`
- Fallback mock response works on any Gemini failure.
- Existing pipeline functionality remains intact.
